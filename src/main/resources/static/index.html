Secure Access Monitoring

I wanted this project to stay simple, but still show a few real security ideas: registration, password checks, account lock after repeated failures, and a protected action that is reviewed in context. The app now decides on its own whether the current browser and source still look familiar for the account.

Password Check

Check the current session

I use this part to see what happens when I confirm the password inside an active session. The app compares the current browser and source with what it already knows for this account.

This step uses the account from the current session: sign in first. I only type the password again. The rest is detected in the background.
Step 3
Protected Action

Try a protected action

Here I pick a real action and let the app judge whether it still looks normal in the current session. The sensitivity of the action is already defined in the backend.

This part also uses the current session: sign in first. I am not typing threat data here anymore. I only choose the action and the app fills in the rest.

Quick checks

I left three short cases here so I can move through the project without setting everything up by hand every time.

1. Same browser

I repeat the password check from the same browser. This is the quiet case.

2. Three bad passwords

I enter the wrong password a few times and let the lockout rule do its job.

3. New browser context

I clear the browser id and ask for data export. The app should treat that combination as more suspicious.

How I read the result

Each response comes back with a short status and a risk level. I read them like this:

  • APPROVED - the request was accepted.
  • DENIED - the password check failed.
  • BLOCKED - the account was locked after repeated failures.
  • REVIEW_REQUIRED - the request was kept for extra review.
